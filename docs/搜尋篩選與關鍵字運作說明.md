# 搜尋流程、篩選流程與關鍵字運作說明

> 最後更新：2026-04-29

---

## 一、排程工作總覽

系統共有 5 個背景排程工作，由 APScheduler 管理：

| 工作名稱 | 函式 | 頻率 | 首次執行 |
|----------|------|------|----------|
| 即時偵測雷達掃描 | `radar_scan` | 每 N 分鐘（預設 5，可設定） | 啟動後 3 分鐘 |
| 市場指標檢查 | `market_check` | 每 60 分鐘（可設定） | 啟動後 3 分鐘 |
| 每日新聞蒐集 | `daily_news_fetch` | 每日 08:00 | 依 cron |
| 每日研究報告蒐集 | `daily_research_fetch` | 每日 10:00 | 依 cron |
| YouTube 頻道偵測 | `youtube_check` | 每 30 分鐘 | 啟動後 3 分鐘 |

> **首次延遲 3 分鐘**：`--reload` 模式下會有兩個 uvicorn process 同時存在，延遲讓舊 process 完全結束，避免重複掃描。

## 二、雷達掃描完整流程（`radar_scan`）

**觸發方式**：自動排程 + 使用者手動觸發（`POST /api/radar/scan`）

```
觸發掃描（自動排程 或 手動強制）
  │
  ├─ Step 1a  RSS / social 來源
  │            asyncio.gather() 並行抓取
  │            → 來源有關鍵字：用來源關鍵字篩選（同時聯集雷達關鍵字）
  │            → 來源無關鍵字：用雷達關鍵字（全局）篩選
  │            → fetch_all=True 的來源：跳過關鍵字篩選，全部進入
  │
  ├─ Step 1b  公開觀測站 MOPS（type="mops" 來源）
  │            → 無任何關鍵字篩選，重大訊息全部通過
  │
  ├─ Step 1c  鉅亨網 API（type="website" 來源）
  │            → 同 Step 1a 的篩選邏輯
  │
  ├─ Step 2   Google News（依 RSS-only 設定決定是否執行）
  │            ├─ rss_only=true → 完全跳過
  │            ├─ RSS 文章數 ≥ radar_rss_min_articles → 跳過
  │            └─ rss_only=false → 依 radar_topics / radar_topics_us 逐詞搜尋
  │
  ├─ Step 3a  主題交叉比對（Pass A2，RSS-only 模式）
  │            → 當 _skip_gn=True 時，對已抓回的 RSS 原始池做主題布林過濾
  │            → 補抓通過 RSS 但未被雷達關鍵字抓到的主題文章
  │
  ├─ Step 3b  主題 Google News 搜尋（Pass B）
  │            → 當 _skip_gn=False 時，每個 Topic 各自發 Google News 查詢
  │            → 每個主題最多 10 個查詢，並發 3 個
  │            → gn_critical_only=true 時，非緊急結果從雷達丟棄（仍存主題頁）
  │
  ├─ 三層去重
  │   ├─ 記憶體：URL + 標題精確比對（seen_urls / seen_titles set）
  │   ├─ 資料庫：查詢 Article 表（URL / 標題）
  │   └─ 內容指紋：Jaccard 相似度 ≥ 0.65（⚠️ 目前暫時停用）
  │
  ├─ 文章評分（_compute_article_scores）
  │   → 計算 decay_factor、novelty_score、finance_relevance、
  │     intensity_score、composite_score 五個欄位
  │
  ├─ 財經相關性篩選（選用，預設關閉）
  │   → finance_filter_enabled=true 且分數 < threshold(0.15) → 丟棄
  │   → fetch_all 來源豁免（一律通過）
  │
  └─ 建立 Alert → WebSocket 廣播 → LINE/Discord/Email 通知
```

### Alert 去重機制

**dedup_key 格式**：`scan:{YYYYMMDDHH}:{md5(first_title)[:16]}`

- 同一小時相同頭條只建一筆 Alert（`Alert.dedup_key` 有 UNIQUE 約束）
- 手動掃描用分鐘精度 `scan:manual:{YYYYMMDDHHmm}:...`，可在同一小時重複觸發

**跨進程鎖**：`SystemConfig["radar_scan_lock"]` 兩次掃描間隔不得低於 240 秒

## 三、來源關鍵字 vs 雷達關鍵字

```
每個 RSS 來源
  │
  ├─ 來源有設關鍵字 → 用「來源關鍵字」篩選
  │                   同時也用「雷達關鍵字」補充（聯集，避免遺漏）
  │
  ├─ 來源無設關鍵字 → 改用「雷達關鍵字」篩選
  │                   若兩者都沒有 → 該來源的文章全部丟棄
  │
  └─ fetch_all=True → 跳過關鍵字篩選，全部文章進入雷達
                      （關鍵字徽章使用「來源關鍵字 + 雷達主題關鍵字」聯集標記，
                       AND 條件須完整命中才顯示；若無任何關鍵字命中則不顯示徽章）
```

## 四、關鍵字篩選邏輯

篩選在**本地**執行，對每篇文章的**標題 + 內文（前 400 字）**做字串比對，不呼叫任何 API。

### 4.1 單一關鍵字（OR 關係）

清單中任一詞命中即通過：

```
關鍵字清單：["通膨", "降息", "衰退"]
→ 文章含其中任何一個即通過
```

### 4.2 空格 = AND

沒有括號時，空格分隔的詞視為 AND 條件（每個詞都要出現）：

```
關鍵字：「信用評等 調降」
→ 文章必須同時出現「信用評等」和「調降」
```

### 4.3 布林分組（AND of OR）

用括號 `()` 劃分群組，括號內用 `OR` 分隔同義詞，**所有括號群組都必須命中**：

```
格式：("Fed" OR "FOMC" OR "聯準會") ("升息" OR "降息" OR "利率")

群組 A：[Fed, FOMC, 聯準會]  → 其中一詞出現在文章
群組 B：[升息, 降息, 利率]  → 其中一詞出現在文章
邏輯：群組 A AND 群組 B
```

**範例：**
```
文章：「FOMC 會議結果：本次維持利率不變」
  群組 A 命中：FOMC ✅  群組 B 命中：利率 ✅  → 通過 ✅

文章：「Fed 主席出席國會聽證會」
  群組 A 命中：Fed ✅  群組 B 命中：(無) ❌  → 過濾 ❌
```

三個以上群組：全部都要同時命中。條件越多，通過率越低，適合精確風險信號。

### 4.4 英文關鍵字

- 大小寫不敏感（自動轉小寫比對）
- **子字串匹配**：`"bank"` 會命中 `"banking"`、`"bankrupt"`
- 需要精確詞邊界時，用完整片語加引號：`"bank run"`

### 4.5 關鍵字徽章（badge）顯示

每篇通過篩選的文章會標記實際命中的詞：

```
topic：("Fed" OR "FOMC") ("升息" OR "降息")
文章包含：FOMC、降息
badge 顯示：FOMC / 降息

最多顯示 4 個詞：
  第一批：每個 AND 群組各取一個代表詞（按群組順序）
  第二批：同群組的其他命中詞填滿剩餘名額
```

> **fetch_all 來源的徽章**：使用「來源關鍵字 + 所有雷達主題關鍵字」的聯集進行標記，AND 條件必須完整命中才顯示。即使來源本身沒有設定關鍵字，雷達主題中命中的關鍵字仍會顯示為徽章。

## 五、主題追蹤關鍵字的 Google News 查詢策略

主題關鍵字同時作用在兩個階段：
1. **Google News 查詢**：組合成搜尋語句取得候選文章
2. **本地二次過濾**：確保 AND 邏輯正確，候選文章仍需通過完整布林比對

### 5.1 單一 OR 群組

簡單列舉（無括號），每個關鍵字**獨立發一次查詢**（並行）：

```
輸入：["Bank run", "Credit default", "Systemic risk"]
→ 各發一次查詢：
  "Bank run"       → 最多 15 篇
  "Credit default" → 最多 15 篇
  "Systemic risk"  → 最多 15 篇
  → 去重合併
```

> 這樣可避免全部合成 1 個 OR 查詢時，冷門詞彙被熱門詞彙搶佔回傳名額的問題。

### 5.2 2 個群組（交叉積 or 錨點）

- 組合數 ≤ 50：交叉積並行查詢

```
群組 A：[市場, 債券, 股市]（3 個詞）
群組 B：[崩盤, 暴跌]（2 個詞）
查詢數：3 × 2 = 6 組
每條：「"市場" "崩盤"」、「"市場" "暴跌"」...
```

- 組合數 > 50：改用**錨點策略**（anchor on 詞彙數較少的群組）

```
群組 A（12 詞）× 群組 B（8 詞）= 96 組 > 50
→ 以 B 為錨點，每條：`"Stagflation" (USA OR China OR EU OR ...)`
實際查詢數：8 組
```

### 5.3 3 個以上群組 → 錨點策略

以**最小群組**為錨點，其餘群組扁平化為 OR：

```
群組 A：[A1, A2]（最小）
群組 B：[B1, B2, B3]
群組 C：[C1, C2, C3, C4]
→ 查詢："A1" (B1 OR B2 OR B3 OR C1 OR C2 OR C3 OR C4)
        "A2" (B1 OR B2 OR B3 OR C1 OR C2 OR C3 OR C4)
查詢數：2 組
```

本地二次過濾仍嚴格執行完整 AND of OR 邏輯。

### 5.4 查詢並發上限

| 場景 | 最大查詢數 | 並發數 |
|------|-----------|--------|
| 主題手動搜尋（Search 頁） | 50（超過用錨點） | 5 |
| 雷達自動掃描（每 N 分鐘） | 10（較保守） | 3 |

## 六、嚴重程度評估

### 6.1 來源固定風險等級（`MonitorSource.fixed_severity`）

使用者可在設定頁為每個來源設定固定風險等級，作為該來源所有文章的**最低風險下限**：

| 設定值 | 行為 | source_w |
|--------|------|----------|
| **緊急** | 直接回傳 critical，不跑動態評估 | — |
| **高風險** | 作為下限 + 注入 source_w = **1.6**（等同官方來源）| 1.6 |
| **低風險** | 僅作為下限，不影響來源可信度 | 原本值 |
| **動態評估**（預設）| 完全由公式決定 | 原本值 |

**設定為「高風險」的效果：**

| 情境 | 計算 | 結果 |
|------|------|------|
| 1 個 high 關鍵字 | 2.0 × 1.6 = **3.2** | high（保底） |
| 1 個 high + 標題內文都有 | 2.0 × 1.6 × 1.3 = **4.16** | **升級為 critical** |
| 2 個 high 關鍵字 | 2.0 × 1.6 × 1.1 = **3.52** | **升級為 critical** |
| 1 個 critical 關鍵字 | 3.0 × 1.6 = **4.8** | **critical** |

> 設計理念：使用者標記來源為「高風險」= 信任該來源，其報導的重要性應被放大。

### 6.2 動態評估（`_assess_severity_single`）

1. **布林規則**（最高優先）：`SystemConfig["severity_rules"]` 中的條件，第一個命中的規則勝出
2. **多維評分**（規則無命中時）：
   - `base_score`：critical 關鍵字命中 +3.0，high 關鍵字 +2.0
   - **否定過濾**：關鍵字前 6 字元內出現否定詞（不/沒/未/不會等）→ 排除
   - **來源可信度加權**：Reuters/Bloomberg ×1.5、官方來源 ×1.6、一般來源 ×1.0；使用者設「高風險」→ 1.6
   - **GN 未知來源懲罰**：來源不在已知 RSS 清單 AND 來自 Google News → ×0.65
   - **標題+內文確認加成**：關鍵字同時出現在標題和內文 → ×1.3
   - **多關鍵字加成**：每多命中一個詞 +0.1，最高 ×1.3
   - 門檻：≥ 3.5 → critical，≥ 2.0 → high，else → low
3. **時間衰減**：超過 `severity_decay_hours`（預設 6h）→ 降一級（critical→high, high→low）

最終風險等級 = **max（動態評估結果, 來源固定下限）**

### 通知門檻

| 頻道 | 門檻 |
|------|------|
| LINE Messaging API | 僅送含 `{critical}` 文章的警報 |
| LINE Bot（使用者主動查詢） | 回覆最新 3 則警報 |
| Discord Webhook | 全部警報 |
| Email (SMTP) | 全部警報 |

## 七、Alert 內容格式

`Alert.content` 是換行分隔的文章清單，每行含嚴重程度前綴：

```
{critical}[Reuters] 台股崩盤 (關鍵字：台股 OR 大盤)
{high}[Bloomberg] 降息預期升溫
{low}[Yahoo Finance] 市場小幅波動
```

`Alert.source_urls` 是 JSON 陣列，每個 URL 同樣有前綴：`{high}https://...`

前端（RadarPage）以 `parseSourceUrl()` 和 `splitArticleLines()` 解析，渲染各文章的嚴重度 badge 並支援按嚴重度過濾。

## 八、文章評分（`_compute_article_scores`）

每篇文章存入 DB 前計算五個評分欄位：

| 欄位 | 計算方式 |
|------|---------|
| `decay_factor` | `exp(-0.1 × 距發布小時數)` |
| `novelty_score` | `1 / (1 + Jaccard ≥ 0.5 的相似文章數)` |
| `finance_relevance` | 本地財經詞 TF-IDF 近似值（`finance_filter.py`） |
| `intensity_score` | `abs((正向詞 - 負向詞) / 總情緒詞)` |
| `composite_score` | `decay × novelty × max(relevance, 0.05) × (0.5 + 0.5 × intensity)` |

### 財經相關性（`finance_filter.py`）

三層詞彙：`FINANCE_CORE`（×3 權重）、`FINANCE_CONTEXT`（×1）、`NON_FINANCE_INDICATORS`（−2 懲罰）

公式：`(core×3 + context − non_fin×2) / sqrt(word_count)`，裁剪到 [0, 1]

**財經篩選**（`finance_filter_enabled=true` 時才啟用，預設關閉）：低於 `finance_relevance_threshold`（預設 0.15）的文章被丟棄。`fetch_all` 來源豁免。

## 九、內容指紋去重

> ⚠️ **目前暫時停用**（`_is_content_duplicate` 直接回傳 `False`）

### 原理

```python
def _article_fingerprint(title, content):
    text = title + " " + content[:500]   # 標題 + 前 500 字
    zh = re.findall(r'[\u4e00-\u9fff]{2,}', text)   # 2 字以上中文詞
    en = [w.lower() for w in re.findall(r'[A-Za-z]{3,}', text)]  # 3 字以上英文詞
    return frozenset(set(zh + en) - _CONTENT_STOPWORDS)  # 去除停用詞

# Jaccard 相似度 ≥ 0.65 → 視為重複
if len(fp & seen) / len(fp | seen) >= 0.65:
    → 丟棄（同一事件不同來源的重複報導）
```

### 適用場景

可捕捉「URL 不同、標題不同，但內容幾乎相同」的重複文章（如 Reuters 原稿被 Yahoo Finance 轉載）。`seen_content_fps` 跨所有來源（RSS → Website → GN → 主題）共享。

## 十、其他排程工作

### 市場指標檢查（`market_check`，每 60 分鐘）

1. 從 `market_data` 服務取得 `MarketWatchItem` 清單的即時報價（yfinance）
2. 依 `priority` 逐一評估 `SignalCondition`（gt / lt / gte / lte / between）
3. **僅在 signal 狀態改變時**才建立 Alert（防止重複）
4. `negative` + 漲跌幅 >5% → `critical`；否則 `high`

### 每日新聞蒐集（`daily_news_fetch`，每日 08:00）

- 固定搜尋詞：`金融市場`、`台股`、`美股`、`經濟數據`、`央行政策`
- 加上所有 `type="rss"` 的 MonitorSource（回溯 24h，每 topic 最多 10 筆）
- 存入 `Article` 表（`category="daily"`）
- 有新文章時自動 append 到 Google Sheets（via GAS Web App）
- **不建立 Alert**；使用者需至「新聞資料庫」頁面手動選取儲存

### 每日研究報告蒐集（`daily_research_fetch`，每日 10:00）

- 資料來源：`MonitorSource` 中 `type="research"` 的所有機構（IMF、BIS、Fed、ECB、BOJ、BOE、NBER）
- 雙模式：`ideas.repec.org` URL → HTML 爬蟲；其他 → 標準 RSS
- 存入 `ResearchReport` 表；**不建立 Alert**，使用者手動至「研究報告」頁選取儲存

### YouTube 頻道偵測（`youtube_check`，每 30 分鐘）

- 對所有 `is_active=True` 的 `YoutubeChannel`，透過 YouTube RSS 取得最新影片
- 新影片存入並標記 `is_new=True`
- 有新影片時廣播 WebSocket `youtube_new_videos`

## 十二、已知限制

- **子字串匹配**：`"降息"` 也會命中 `"不降息"`，中文需注意否定詞（否定過濾在嚴重度評估有處理，但關鍵字篩選階段沒有）
- **三層 AND 組合**：通過率極低，適合精確風險信號，不適合一般新聞訂閱
- **英文部分匹配**：`"bank"` 命中 `"banking"`，精確詞邊界需用完整片語
- **Google News 索引延遲**：事件發生到出現在 GN RSS 最慢約 2 小時，無法靠縮短掃描間隔解決
- **Google News 不完全支援 NOT 語法**：複雜 NOT 操作符效果不保證
- **內容指紋目前停用**：同一則新聞的多來源轉載不再自動去除
